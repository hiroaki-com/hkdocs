The site search had fallen into a state where "new articles wouldn't show up, yet irrelevant articles kept surfacing." After investigating, the cause turned out to be not a lack of Algolia tuning, but a structural problem: **the index the frontend queried and the index the crawler updated were mismatched**.

This article is a record of the process, from diagnosing that cause to restoring accuracy by consolidating into a single index. For the continuously maintained procedure, see the [technical documentation version](/en/docs/tech/docusaurus/docusaurus-algolia-single-index-search-precision/).

This article is a sequel in this site's Algolia adoption and localization series. For the setup steps, see [How to Integrate Algolia DocSearch into Docusaurus](/en/blog/2025/06/10/setup-sitesearch-with-algolia-docsearch/); for the initial attempt at per-language optimization, see [How to Optimize Algolia Search by Language on a Docusaurus Multi-language Site](/en/blog/2025/06/24/docusaurus-algolia-multilingual-search-opt/). This article **corrects and improves** the latter approach into a single-index setup.

### The Challenge: New Articles Don't Appear in Search

This site previously used a setup that split Algolia search into "per-language indexes" (`yoursite_ja` / `yoursite_en`). At some point, however, I began to notice the search accuracy declining.

When I actually ran queries with the public search key to check, the results were as follows:

| Query | Result (Before) |
| :--- | :--- |
| "count up" / "darts" | Only irrelevant exam pages matched |
| "decap" | 0 results |

Newer articles matched nothing at all, while old irrelevant articles surfaced. This was the classic symptom of **querying a stale index**.

### The Cause: The Queried and Updated Indexes Were Mismatched

Cross-checking the configuration revealed the cause.

-   The Algolia setting in `docusaurus.config.ts` was still `indexName: 'yoursite'` (the single name from before the split).
-   Meanwhile, the crawler was only updating the two indexes `yoursite_ja` / `yoursite_en`.
-   As a result, `yoursite` (the single name) had barely been updated since the day of the per-language split, and **about a year's worth of articles was missing**.

The frontend kept querying the `yoursite` index that had stopped being updated, while the crawler kept updating `yoursite_ja` / `yoursite_en`, which no one referenced — a complete mismatch.

:::danger Orphaned indexes are hard to notice
The crawler runs normally, and the per-language indexes are updated daily. The record counts on the dashboard also look healthy, so the single fact that "the frontend isn't referencing them" becomes a black box, and it surfaced only later as a decline in accuracy.
:::

### Root Cause: A Misunderstanding of `contextualSearch`

Why did the mismatch happen? The starting point was a misunderstanding of `contextualSearch: true`.

Initially, I thought "`contextualSearch: true` appends the locale to `indexName` and queries `yoursite_ja` / `yoursite_en`," and I split into per-language indexes on that assumption. **This is wrong.**

Checking the `@docusaurus/theme-search-algolia` source (`useAlgoliaContextualFacetFilters`) and the build output, `contextualSearch` behaves as follows:

-   **It never changes `indexName`.**
-   It merely queries a single index and narrows results at query time with `facetFilters` (`language:<locale>` and `docusaurus_tag:*`).

In other words, filtering by language is achieved with a **single index + facets**, and splitting into per-language indexes was unnecessary from the start. The two split indexes had become "orphaned indexes" that the frontend never referenced.

### The Solution: Consolidate into a Single Index

The approach is simple. **Consolidate the crawler into a single `yoursite` index so that the index the frontend queries matches the index that gets updated.** Since language filtering is handled by the already-enabled `contextualSearch: true` (the `language` facet), no change to `docusaurus.config.ts` is needed.

At the same time as consolidating, I also performed the following noise removal and language tuning.

| Item | Old | New | Purpose |
| :--- | :--- | :--- | :--- |
| indexName | `yoursite_ja` + `yoursite_en` | Single `yoursite` | Match the queried and updated indexes to restore freshness |
| exclusionPatterns | None | Exclude tags / authors / category / page / search, etc. | Remove noise from thin pages |
| content | `p, li, td` | `+ article pre` | Include code blocks in search |
| ignorePlurals | ja=false / en=true | `["en"]` | Treat plurals as identical for English only, in a single index |
| indexLanguages / queryLanguages | None | `["ja", "en"]` | Dictionary segmentation for Japanese (improves compound-word accuracy) |
| minWordSizefor1Typo / 2 | 3 / 7 | 4 / 8 | Suppress excessive typo matches in Japanese |

### Applying It: The UI Was Unavailable, So I Updated Directly via the Crawler REST API

The tricky part was the path to applying the changes. The "Review and publish" button in the Algolia Crawler admin UI was unusable because the app's **Domain Verification was disabled (the DocSearch Status field was blank)**.

So I applied the changes by **updating the production config directly via the Crawler REST API**. This path bypasses the UI gate, and the existing scheduled crawls run on this API configuration as well.

```bash
# Auth: --user "CRAWLER_USER_ID:CRAWLER_API_KEY"
# Get config:  GET   https://crawler.algolia.com/api/1/crawlers/{crawlerId}?withConfig=true
# Update config: PATCH https://crawler.algolia.com/api/1/crawlers/{crawlerId}/config
# Re-crawl:    POST  https://crawler.algolia.com/api/1/crawlers/{crawlerId}/reindex
# Index settings: PUT https://{APP_ID}.algolia.net/1/indexes/yoursite/settings
```

The operations performed were three steps:

1.  `PATCH .../config` … Consolidate `actions` into a single `yoursite` and add `exclusionPatterns`.
2.  `PUT .../indexes/yoursite/settings` … Apply `indexLanguages` / `queryLanguages` / `ignorePlurals` / typo tolerance, etc.
3.  `POST .../reindex` … Run a re-crawl before and after applying the settings.

:::warning Two gotchas
*   `recordExtractor` must be sent in the `{"__type":"function","source":"..."}` format. Sending it as a plain string results in a `validation_error`.
*   `initialIndexSettings` is **not automatically applied** to an existing index. To reflect the settings, you must inject them directly with `setSettings` (`PUT .../settings`).
:::

:::danger Do not publish from the UI
The Crawler UI editor still contains the old code from the split era, and publishing from there rolls back the improvements. Make all future configuration changes via the API, and since the Crawler API / index keys used for the work have admin privileges, it is safest to rotate them afterward.
:::

### Result: Accuracy Clearly Restored

Re-verifying with the same public search key, the Before → After difference was clear.

| Query | Before | After |
| :--- | :--- | :--- |
| count up | Only exam pages | The intended article ranks at the top ✅ |
| darts / darts calculation | Only exam pages | The article group ranks high (compound words match via dictionary segmentation) ✅ |
| decap | 0 results | The intended article ranks at the top ✅ |
| `/blog/tags/*`, `/diary/*` | Mixed in as noise | 0 results (successfully excluded) ✅ |

After applying the language tuning, the number of hits was appropriately narrowed, and irrelevant partial matches in Japanese also decreased.

### Summary

*   For declining search accuracy, before tuning, suspect a mismatch between **"the index the frontend references" and "the index the crawler updates."** When these diverge, you keep querying a stale index behind a crawler that looks healthy.
*   Docusaurus's `contextualSearch: true` does not change `indexName`. Language filtering is achieved with a **single index + the `language` facet**, and splitting into per-language indexes is generally unnecessary.
*   Even with a single index, per-language tuning is entirely possible via **language-array specifications** such as `ignorePlurals: ["en"]` and `indexLanguages: ["ja", "en"]`.
